# SLPVMFW Mask Build

## Repository Layout

Resolve the repository portably. From a sibling `oneos-sim` checkout, the default layout is:

```text
git/
├── oneos-sim/
└── slpvmfw/
```

Use `../slpvmfw` from `oneos-sim`, or discover the `slpvmfw` git root from the current directory. Do not hardcode `/home/tring/...` in commands or explanations.

Important paths inside `slpvmfw`:

- Generated mask C: `build/mask/qzvm_mask.c`
- Build file: `build.xml`
- Common build rules and environment variables: `build_common.xml`
- Main Java framework sources compiled by `build.xml`: `card/`, `org/globalplatform/`, `uicc/`, `com/cmcc/`, `whiteCard/write/personal/`
- API/declaration sources used by `build_api.xml`: `api/`
- HCI framework examples: `uicc/hci/framework/` and `api/uicc/hci/framework/`

## Branch Mapping

Before searching Java code, explaining generated mask C, editing framework Java, or building, ensure the `slpvmfw` branch matches the active `oneos-sim` branch. The `oneos-sim` repo is usually a sibling of `slpvmfw`; use the current `oneos-sim` branch unless the user explicitly names a COS branch.

Mapping:

| oneos-sim branch | COS partition | version | slpvmfw framework branch |
| --- | --- | --- | --- |
| `dev_4.0` | `cosA` | `010000` | `cosup_COSA_V1` |
| `dev_4.0_COSA` | `cosA` | `010300` | `cosup_COSA_V1` |
| `dev_4.0_COSB` | `cosB` | `010200` | `cosup_COSB_V1` |
| `dev_4.0_COSC` | `cosA` | `010400` | `cosup_COSC_V1` |
| `COS_V2` | `cosB` | `010000` | `cosup_COS1_V2` |
| `COS_1_V2` | `cosB` | `010300` | `cosup_COS1_V2` |
| `COS_2_V2` | `cosA` | `010200` | `cosup_COS2_V2` |
| `COS_3_V2` | `cosB` | `010400` | `cosup_COS3_V2` |

If the active `oneos-sim` branch is not in this table, default the framework branch to `cosup_COSA_V1` and state that the default mapping was used.

Required preflight:

```bash
git -C ../oneos-sim branch --show-current
git -C ../slpvmfw branch --show-current
```

If `slpvmfw` is on the wrong branch, switch it before analysis or build. Prefer an existing local branch:

```bash
git -C ../slpvmfw switch <mapped-framework-branch>
```

If the local branch does not exist but the remote branch exists, create the local branch from the remote branch:

```bash
git -C ../slpvmfw checkout -b <mapped-framework-branch> origin/<mapped-framework-branch>
```

Example:

```bash
git -C ../slpvmfw checkout -b cosup_COS3_V2 origin/cosup_COS3_V2
```

If neither the local branch nor `origin/<mapped-framework-branch>` exists, stop and report the missing framework branch.

Do not switch branches if `git -C ../slpvmfw status --short` shows local changes that are not yours or not understood. In that case, stop and report the mismatch plus the dirty files.

## Hard Rule

Never manually edit `build/mask/qzvm_mask.c` or other files under `build/mask/` to implement behavior changes. Those files are generated by the Ant/QZVMDK build. If the user asks to modify qzvm_mask.c behavior, find and edit the owning Java source or build/config source, then rebuild.

You may read `build/mask/qzvm_mask.c` for analysis, symbol lookup, diffs, and verification.

## Explaining Mask Symbols

When the user asks what a symbol, function, table, constant, or generated block in `qzvm_mask.c` does:

1. Run the branch preflight and ensure `slpvmfw` is on the mapped branch.
2. Locate the symbol in `build/mask/qzvm_mask.c`.
3. Search Java framework sources for matching class names, package names, method names, field names, string constants, AIDs, and related generated identifiers.
4. Correlate the generated C back to the Java source package/class/method. Prefer `rg` over slower search tools.
5. Explain both sides:
   - what the generated C symbol appears to represent
   - which Java source(s) define the behavior
   - how the Java semantics map into the generated mask artifact
6. If the C symbol cannot be confidently mapped to Java, state the uncertainty and show the closest evidence.

Useful search starts:

```bash
rg -n "<symbol-or-fragment>" build/mask/qzvm_mask.c
rg -n "<class-or-method-or-aid-fragment>" card org uicc com api whiteCard
```

## Editing Workflow

For behavior changes that mention `qzvm_mask.c`, `mask.c`, or generated mask output:

1. Run the branch preflight and ensure `slpvmfw` is on the mapped branch.
2. Inspect the generated C only to understand the output and identify candidate Java owners.
3. Edit Java source under the framework source tree, or build/config files only when the requested change is a build/config change.
4. Do not edit `build/mask/qzvm_mask.c` directly.
5. Build from the `slpvmfw` repo root:

```bash
ant clean
ant
```

6. Verify that `build/mask/qzvm_mask.c` was regenerated and inspect the relevant diff/output.
7. Report the oneos-sim branch, framework branch, Java files changed, build result, and generated mask file location.

## Build Notes

`build_common.xml` requires environment variables such as `QZVM_HOME` and `QZVM_SDK_HOME`; `XCTOOLS_EXE` and `QZVM_QZRT_HOME` may override tool locations. If `ant clean` or `ant` fails because these are missing, report the exact missing variable or tool path rather than guessing.

Run build commands from the `slpvmfw` root:

```bash
cd ../slpvmfw
ant clean
ant
```

The default Ant target is `all`, which compiles Java, converts packages, generates `build/mask/qzvm_mask.c`, and installs mask files into the configured QZRT tree.

## Safety

Check `git status --short` before editing. Preserve unrelated user changes. After building, note generated-file changes separately from Java/source changes.
